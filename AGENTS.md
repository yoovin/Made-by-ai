# Made by AI 운영 지침

## 1. 프로젝트 정체성

- 이 프로젝트는 **실서비스 웹 플랫폼**이다.
- 서비스 구조는 **허브형 제품 구조**다.
- 사용자는 항상 첫 엔트리(허브)에서 서비스 카드/링크/버튼을 통해 각 기능으로 이동한다.
- 새 기능은 "새 사이트"가 아니라 "허브 안에 추가되는 새 서비스"로 취급한다.

## 2. OpenCode 실행 방식

- OpenCode는 두 방식으로 실행된다.
  1. 사용자가 직접 TUI를 여는 수동 세션
  2. `scripts/autoloop.sh`가 `opencode run`으로 돌리는 자동 세션
- 이 저장소의 규칙은 수동/자동 세션 모두에 항상 적용되어야 한다.
- 진행 알림은 `.env.local`의 Telegram 설정을 사용한다.

## 3. 필수 구조 규칙

- 사용자에게 노출되는 모든 기능은 반드시 `src/lib/service-registry.ts`에 등록한다.
- 사용자 접근 경로 없이 고아 페이지를 만들지 않는다.
- 새 기능을 만들면 아래를 반드시 같이 수행한다.
  1. 라우트 또는 화면 생성
  2. service-registry 등록
  3. 허브 화면에서 접근 가능하게 연결
  4. changelog 기록

## 4. 실행 환경

- 실행 환경은 Docker가 아니라 **로컬 Ubuntu**다.
- GitHub Actions는 사용하지 않는다.
- GitHub는 **버전 관리용 원격 저장소**다.
- 로컬 배포/실행은 현재 로컬 Ubuntu 기준으로 수행한다.
- 실서비스 외부 포트는 **80번 포트**다.
- 애플리케이션 내부 실행 포트는 기본적으로 `3000`을 사용하고, nginx가 `80 -> 3000`으로 프록시한다.
- 배포 관련 변경은 반드시 `scripts/deploy-local.sh`, `scripts/smoke-test.sh` 흐름을 우선 사용한다.

## 5. OpenCode / oh-my-opencode 원칙

- 이 프로젝트는 OpenCode + oh-my-opencode 환경을 전제로 한다.
- 별도 `.env` API 키를 요구하지 않는다.
- 로컬 OpenCode에 이미 연결된 인증 정보로 동작한다고 가정한다.
- 가능한 한 Plan → Build 흐름을 따르되, 작은 안전 작업은 바로 수행할 수 있다.
- 사람이 읽을 가치가 있는 개발 흐름은 Telegram으로 한국어 요약이 전송되도록 유지한다.

## 6. 기본 개발 루프

사용자가 어떤 작업을 지시하면 항상 아래 순서를 우선한다.

1. 현재 상태 파악
2. backlog / approval queue / changelog / service-registry 확인
3. 가장 안전하고 가치 있는 작은 작업 선택
4. 구현
5. 검증
6. 허브 연결 여부 점검
7. changelog 업데이트
8. 로컬 배포 필요 여부 판단
9. 배포 시 smoke test
10. git commit / push 필요 여부 판단
11. Telegram 진행상황 보고

## 7. 우선순위

항상 아래 순서로 우선한다.

1. 프로덕션 장애/오류
2. 데이터 무결성/보안 문제
3. 허브 진입 및 핵심 흐름 문제
4. 성능/안정성 개선
5. UX 명확성 개선
6. 사용자 가치가 높은 작은 기능 추가
7. 리팩터링

## 8. 위험 변경 정책

아래에 해당하면 **위험 변경**이다.

- 인증 / 권한 / 세션
- 결제 / 구독 / 과금
- 파괴적 DB 변경
- 데이터 삭제 / 대량 수정 / 백필
- 개인정보 / 시크릿 / 보안 민감 코드
- nginx / reverse proxy / 포트 / 서비스 매니저 / 배포 스크립트 큰 변경
- 외부 서비스 신규 연동
- 실서비스 80번 포트 트래픽에 직접 영향을 주는 구조 변경

위험 변경이 필요하면:

1. 즉시 배포하지 않는다.
2. `ops/approvals/pending/<id>.json` 파일을 생성한다.
3. 파일에는 제목, 요약, 이유, 영향 범위, 리스크, `resumePrompt`, `discardPrompt`를 구체적으로 적는다.
4. Telegram 승인 봇이 사용자와 소통하도록 둔다.
5. 해당 위험 작업은 승인 전까지 보류한다.
6. 같은 세션 안에서 할 수 있는 다른 안전 작업이 있으면 계속 진행한다.
7. 승인되면 재개하고, 폐기되면 정리 후 대안 작업으로 넘어간다.

## 9. 검증 규칙

배포 전 가능한 범위에서 아래를 수행한다.

- lint
- typecheck
- build
- health check
- 핵심 화면 smoke test

테스트가 없으면 거짓으로 "테스트 완료"라고 쓰지 않는다.
실제로 수행한 검증만 changelog에 기록한다.

## 10. 배포 규칙

- 로컬 배포는 `scripts/deploy-local.sh`를 우선 사용한다.
- 배포 후 `scripts/smoke-test.sh`를 실행한다.
- smoke test 실패 시 즉시 실패로 보고하고 원인 정리를 남긴다.
- 되돌릴 수 있는 작은 변경을 우선한다.

## 11. Git 규칙

- GitHub는 버전 관리용이다.
- 안전한 배포가 끝났고 changelog가 업데이트되면 commit / push를 수행한다.
- 의미 없는 거대 커밋보다 작은 커밋을 선호한다.
- 커밋 메시지는 구체적으로 작성한다.

## 12. Telegram 보고 규칙

- 진행 흐름을 한국어로 요약한다.
- 너무 시시한 로그를 모두 보내지 말고, 사람이 읽을 가치가 있는 변화 중심으로 보낸다.
- 최소한 아래는 알려야 한다.
  - 세션 시작
  - 현재 작업 목표
  - TODO 큰 변화
  - 중요한 파일 수정
  - 테스트/빌드/배포 시작과 결과
  - 오류/중단
  - 승인 요청
  - 세션 종료 요약

## 13. 체인지로그 규칙

모든 배포 또는 배포 시도는 `ops/changelog/CHANGELOG.md`에 기록한다.

항목에 반드시 포함할 것:

- 시각
- 한 줄 요약
- 변경 이유
- 사용자 영향
- 기술 변경
- 검증 내용
- 리스크 / 제한사항

## 14. 절대 금지

- 허브에 연결되지 않는 사용자용 페이지를 무단 공개
- 승인 없는 위험 배포
- 검증하지 않은 내용을 검증했다고 주장
- `.env.local` 등 민감정보를 Git에 커밋
- 노출된 Telegram 토큰 재사용


## 15. 자동 루프 규칙

- 자동 루프는 한 사이클마다 가장 안전하고 가치 있는 작은 작업 1개만 수행한다.
- 승인 대기 작업이 생겨도 전체 루프를 멈추지 않는다.
- pending approval에 올라간 작업은 보류하고, 다른 안전 작업을 계속 찾는다.
- 각 사이클 종료 시 `ops/last-summary.md`에 요약을 남긴다.
- 자동 push는 환경변수 `LOCAL_GIT_PUSH=true`일 때만 허용한다.
